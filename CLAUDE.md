# CLAUDE.md

Guidance for working on **djangofmt** — a fast, HTML-aware Django/Jinja template formatter and
linter, written in Rust. See `CONTRIBUTING.md` for environment setup (`just bootstrap`) and the
AI-contribution policy.

## Workspace

- `crates/djangofmt` — the CLI: formatting is the default command, `check` is the lint subcommand.
- `crates/djangofmt_lint` — the linter (rules, registry, checker/visitor).
- `crates/djangofmt_dev` — documentation generation (`cargo run -p djangofmt_dev -- generate-all`).
- `crates/djangofmt_wasm`, `crates/djangofmt_macros`, `playground/` — wasm bindings, proc-macros, web playground.
- Parsing and formatting come from a **fork** of `markup_fmt` (pinned `rev` in `Cargo.toml`); the
  lint crate works on top of its `markup_fmt::ast`.

## Verify before committing

Run `just pre-mr-check` until green — it runs the pre-commit hooks, `maturin develop`, clippy, the
test suite, and a strict docs build. For lint-only work, `cargo test -p djangofmt_lint` then
`cargo insta accept` to update snapshots.

Workspace clippy denies `pedantic` + `nursery` and `unwrap_used` (`Cargo.toml [workspace.lints]`) —
keep it warning-clean.

## Generated docs are not committed

`docs/index.md`, `docs/contributing.md`, `docs/rules.md`, and `docs/rules/*` are generated by
`just docs-generate` and are gitignored (`docs/.gitignore`). **Never commit them** — the strict
docs build regenerates and validates them from source.

## Linter architecture (`djangofmt_lint`)

A rule is a violation struct plus a `check` function, wired into a single-pass AST visitor:

- The struct's `///` docstring **is** the rule's published documentation (rendered to
  `docs/rules/<slug>.md` via `#[derive(ViolationMetadata)]`). Follow the
  `## What it does` / `## Why is this bad?` / `## Example` layout.
- `impl Violation` sets `RULE`, `CATEGORY` (correctness / suspicious / style / complexity /
  accessibility), optional `FIX_AVAILABILITY`, and `message` / `help`.
- Register in `define_rules!` (`registry.rs`), then wire the `check` call into the right `visit_*`
  method in `checker.rs`, gated by `is_rule_enabled`.
- Report via `checker.report_diagnostic(&violation, (offset, len).into())`; get byte offsets with
  `checker.source_offset(slice)`.

**Cross-node rules** (those that decide from many nodes — duplicate names, tag ordering) must not
re-walk the tree. Mirror ruff's deferred analysis: accumulate state into a `Checker` field during
the single existing pass, then run a finalize `check(checker)` after the `visit_root` children loop.
To store borrowed `&'a str` slices from the AST, thread the source lifetime `'a` through the
`visit_*` signatures. Example: `rules/correctness/duplicate_block_name.rs`.

**Ruff alignment**: borrow ruff's idioms where they fit this project's scale — deferred analysis,
positions/offsets as the currency (re-slice the source for text), all-but-first deduplication. Do
**not** cargo-cult its scale-driven machinery (semantic model, string interning, typed indices); at
this rule count it is overkill.

## Tests

Rule fixtures live in `crates/djangofmt_lint/tests/check/<rule>/<rule>.{valid,invalid}.html`;
diagnostics are snapshotted with `insta`. The runner lints every fixture with **all rules enabled**
(`Settings::all()`), so `valid.html` must produce zero diagnostics against *every* rule, not just
the new one — e.g. a `{% blocktranslate %}` guard case needs `trimmed`, an `<html>` case needs
`lang`.

## Conventions

- **Adding a lint rule**: use the `add-lint-rule` skill (`.agents/skills/add-lint-rule/`). Land it as
  a single commit titled ``Add `<slug>` lint rule`` with the rule's docstring as the body.
- Do non-trivial work in an isolated git worktree; commit only once `just pre-mr-check` is green;
  push only when asked.
